Android FragmentTabHost 的一个bug

该官方类文档

bug发现过程

bug是这样的,官方提供的demo的widgetTab是在顶部的, 我想让它在底部= =然后 不管怎么更改布局界面它都不会下来
我在xml里加入了自己的子View发现根本不会出现, 之前看过tabHost的源码, 要是没有指定特定的view id值, 则会自己新建一个布局;

直接看FragmentTabHost的源码, 发现在构造函数中调用了initFragmentTabHost 函数,
然而在此函数中, 有这么一个判断


好吧原因已经找到了,就是这个null 判断一直是成立的导致他根本没用我们自定义的布局, 一直使用默认布局,
所以tabWidget 就一直都是在底部了, ok 我们很容易就能想到 肯定是由于顺序问题 此时的xml 布局还没有加载到该viewGroup中, 所以此时取就是一直是一个null了! = =
由于当时项目就要用这个类, 所以我只能重新编译了这个jar包了, 导入源码后注释掉该判断 这段代码
(省略1000字, 移动了很多文件, 还有些莫名的错误看了下好像不怎么可能用到, 就直接注掉了) 重新编译 ok 可以用了!!!

~~ ————- 好的我们来分析下导致这个bug的原因,

首选看demo中 mFragemntTabHost = (FragmentTabHost)inflater.inflate(R.layout.fragment_tab_host, viewGroup);
FragmentTabHost 是被这样实例化出来的 然后 id 为 android.R.id.tabs 被包含在 R.layout.fragment_tab_host 的xml中! 好的我们可以写个简单的demo 来看看xml中的view 是是何时依附在viewGroup中的
(ps:我猜应该是在 onFinishInflate 因为看名字像! ^-^~~
看下官方文档:

Finalize inflating a view from XML. This is called as the last phase of inflation, after all child views have been added.

Even if the subclass overrides onFinishInflate, they should always be
sure to call the super method, so that we get called.
好吧绝对是在这之后鸟,看在是google的bug,
还是写个简单的demo 验证下吧!!!

我们只要验证一下继承ViewGroup类, 就继承FrameLayout吧= = ! 打印了下各个几个时期下 是否取到了xml中的id
